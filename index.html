Dolt sync, explained

What this is: a walkthrough of how Dolt — a SQL database that versions like Git — syncs through a remote, with seven reproducible shell-script scenarios you can run yourself. Driven by a concrete use case: syncing beads, the task DB, across sessions and machines.
Repo: github.com/idvorkin-ai-tools/dolt-explainer · Scenarios live in: scripts/
TL;DR. Dolt pushes a SQL database through any git remote — including a plain GitHub repo — by writing its chunks to a hidden ref refs/dolt/data. GitHub stores the blobs happily; GitHub's web UI doesn't render them (it only shows refs/heads/*). So a Dolt-backed GitHub repo looks empty in the browser but round-trips a full database via dolt clone.

The illusion

Dolt ships with its own remote protocol, but when you point it at a URL ending in .git it switches into git mode: it speaks the Git Smart HTTP protocol, uses your ordinary git credential helper, and pushes all its database state to a sibling ref called refs/dolt/data. A Dolt push looks like this on the wire:

$ dolt remote add origin https://github.com/owner/repo.git
$ dolt remote -v
origin  git+https://github.com/owner/repo.git
#       ^^^^ dolt auto-added this prefix
$ dolt push -u origin main

What GitHub stores afterward, asked via git ls-remote:

$ git ls-remote https://github.com/owner/repo.git
d4cc9f8…  HEAD
0ede5af…  refs/dolt/data      <- the entire database
d4cc9f8…  refs/heads/main     <- just the seed README
Aha. GitHub's file browser renders only refs/heads/main. If main holds a single README commit, that's all a visitor sees. The chunks on refs/dolt/data are a tree of opaque files — manifest, a .darc archive, and content-addressed Noms chunks — that git stores faithfully but GitHub has no UI for.

A vanilla git clone fetches only refs/heads/* and gets the README. A dolt clone fetches refs/dolt/data and rebuilds the database. Same URL, two realities.

Want to see it on the real GitHub? The reference repo idvorkin-ai-tools/dolt-sync-test is kept as a permanent demo — look at the file browser (empty-ish), then git ls-remote the URL and spot the hidden ref.

Conflicts are SQL rows

When two clones edit the same row, Dolt materializes the conflict as actual rows in a table called dolt_conflicts_<your_table>, with base_* / our_* / their_* columns. You resolve by querying and committing:

$ dolt sql -q 'SELECT * FROM dolt_conflicts_items;'
$ dolt conflicts resolve --theirs items    # or --ours, or write SQL
$ dolt commit -am 'merge: keep their value'

That's arguably the biggest conceptual upgrade from git: conflict resolution becomes a database operation rather than a text-file merge.

The fork-workflow problem

If you use the idvorkin / idvorkin-ai-tools split — canonical upstream owned by a human, fork writable by AI tools, landings go via GitHub PR — Dolt's ref layout introduces a hole the plain Git workflow doesn't have.

Three things all go wrong at once:

  1. dolt push doesn't update refs/heads/main. A Dolt push writes to refs/dolt/data only. The seed commit on refs/heads/main stays put, on both fork and upstream, from the moment the repo was created.
  2. GitHub's PR diff shows "nothing to compare." The compare view diffs refs/heads/<base>..refs/heads/<head>. Both sides are the same seed commit, so GitHub says the PR is empty — even though the fork's database differs from upstream's.
  3. GitHub's "Merge PR" button only advances refs/heads/main. It can't move refs/dolt/data. Clicking Merge changes nothing for the database. The data stays on the fork; upstream's refs/dolt/data is still whatever it was before the PR was opened.

The only path that actually lands AI-tools' data into upstream is one the AI tools can't take, because they don't have upstream write:

# Human admin with upstream write runs locally:
dolt remote add fork https://github.com/idvorkin-ai-tools/dolt-thing.git
dolt fetch fork main
dolt merge fork/main
dolt push upstream main       # this push carries refs/dolt/data to upstream

If you want a fork-model where AI tools can fully self-serve, the options are:

  • Ship AI tools upstream write (defeats the fork boundary).
  • Add a GitHub Action on upstream that reacts to PR-merged and pulls refs/dolt/data from the fork. Doable, but you're writing custom glue every time.
  • Use DoltHub instead of GitHub as the remote — it has dolt-aware pull requests that diff tables and merge refs/dolt/data when you click Merge.

Scenario 7 below reproduces all three failure modes locally with two bare git repos standing in for upstream and fork.

Why we use Dolt anyway: beads sync

The fork/PR model is fit for authored content — prose, code, skills — where a human adds value by reviewing line-level diffs and deciding whether the change lands. But the ai-tools ecosystem also produces operational data: task state written by agents, read by other agents, no human author. Beads (bd) is the canonical example — it stores issues in a Dolt database and syncs with bd dolt push / bd dolt pull. Every session on every machine writes to it; every session reads from it.

The pattern: one Dolt-backed repo under idvorkin-ai-tools/, no upstream, every agent pushes and pulls directly. That inverts the blog-code workflow deliberately, and it's the inversion the fork-hole analysis above pushes you toward.

Why this model fits task data (and wouldn't fit a blog post)

Blog code (authored, upstream + fork)Beads data (operational, ai-tools only)
Prose and code — needs human reviewTask state — written by agents, not authored
Line-level diffs a human reads on GitHubRow-level diffs nobody is going to review on a PR ("bd-123 in_progress → completed")
One writer at a time via PR mergesMany concurrent writers — every session on every machine
Conflicts are rare and human-resolvedConflicts would be constant if serialized through PRs — Dolt's row-level three-way merge makes them tolerable without a reviewer

Why Dolt specifically (not SQLite-in-git, not Postgres)

  1. Mergeable by rows, not bytes. A SQLite file in git is an unmergeable binary blob — two sessions writing at once = immediate conflict mess. Dolt merges at the row level (see scenario 4), so concurrent agents almost always merge clean.
  2. No server, no credentials. Just GitHub as a dumb chunk store. bd dolt push is git push with a SQL database onboard. No RDS, no Postgres instance, no secret rotation.
  3. Audit trail for free. dolt log + dolt diff give per-row history. Which session closed bd-123? When? From what state? Answerable.
  4. Offline-first. Sessions commit locally; push when online. Network blips don't break agents mid-write.

The trade accepted: no human PR review of beads changes. Task state isn't authored content, so that's fine. Anything that IS authored — blog posts, skills, code — stays on the git + fork + PR model.

Scenarios

Six reproducible shell scripts. Each wipes its own run directory, sets up fresh state, and leaves artifacts for you to poke at. ./scripts/run-all.sh runs them end-to-end in ~20 seconds, offline.

1. Bootstrap — init, table, commit, log, query

Dolt's basic “git for tables” loop. Init a database, create a schema, insert rows, stage and commit. dolt log and dolt status behave like their git counterparts; the data is stored in Dolt's own chunk format, not as SQL text.

2. File remote — push → file:// → clone into a second dir

A Dolt remote is a directory of content-addressed chunks. dolt push materializes them on disk; dolt clone pulls them into a new working copy. No server required.

3. Round-trip — edit in B, pull in A

Clone B adds a row and pushes. Clone A pulls and sees the row. History preserves B's author/message on A's side — same as git.

4. Clean merge — concurrent inserts on different rows

A and B each insert a different row. First push wins; second hits non-fast-forward. dolt pull runs a three-way merge with no human input — the result has a merge commit and both rows.

5. Conflict — concurrent edits to the same row

Merge stops. The conflict is stored as a row in dolt_conflicts_items with base_* / our_* / their_* columns. Resolve with dolt conflicts resolve --theirs (or write SQL), commit the merge, push.

Running the scenarios yourself

git clone https://github.com/idvorkin-ai-tools/dolt-explainer.git
cd dolt-explainer
brew install dolt                    # or the user-local installer
./scripts/run-all.sh                 # ~20 s, offline, writes to runs/
./scripts/run-all.sh --live          # also run 06b against real GitHub (needs gh auth)
