Beads + Git sync in one fork repo

The problem: agents on multiple machines need to share task state (beads) in near-real-time, while the underlying blog is on the idvorkin / idvorkin-ai-tools fork workflow for code. Two data classes, one repo, different cadences.
The trick: Dolt pushes the task DB to a sibling ref (refs/dolt/data) on the same GitHub repo that holds code on refs/heads/main. Git and Dolt never collide because they write disjoint refs.
Repo: github.com/idvorkin-ai-tools/dolt-explainer · Scenarios: scripts/
TL;DR. idvorkin-ai-tools/<repo>.git carries two refs: refs/heads/main for code (flows through PR to idvorkin/<repo>) and refs/dolt/data for the beads task database (fork-only, many writers, merged row-by-row). The daily ritual git pull --rebase && bd dolt push && git push is exactly this dual-ref choreography.

The problem we're solving

Constraints.
  • Agents create, close, and mutate beads issues constantly — every session, on whatever machine. Task state must visibly sync within minutes.
  • The blog has an authoritative upstream (idvorkin/idvorkin.github.io) that only a human updates via reviewed PRs from a fork (idvorkin-ai-tools/idvorkin.github.io).
  • We don't want a second repo for beads. Tasks belong with the code they reference — searchable together, clonable together.
  • Beads issues aren't authored prose. No human is going to review a PR that says "bd-123 status: in_progress → completed." They're operational state, not content.

Put all four together and you get a constraint that forces the design: task state must flow freely between agents without a PR review step, while code must keep flowing through PR review, and both must live in the same repo.

That's what the refs/dolt/data trick enables. Dolt pushes its database to a private ref on the same git repo that GitHub uses for code. GitHub's web UI (and git clone) only renders refs/heads/*, so the task DB is invisible to human workflows like PR review — exactly what we want — but every agent with a clone can bd dolt push/pull to stay in sync.

One repo, two refs, two workflows

Code and beads cohabit one GitHub repo because they write different refs:

Data classRefWho writesWhere it flowsHow it merges
Blog code & proserefs/heads/main, refs/heads/<branch>You, via git pushFork → upstream via PRHuman review + git merge / rebase
Beads task DBrefs/dolt/dataEvery agent on every machine, via bd dolt pushFork only — never flows to upstreamDolt row-level three-way merge; conflicts become queryable SQL rows

The daily workflow

The blog's CLAUDE.md session-completion ritual:

git pull --rebase        # pull others' code onto my local main
bd dolt push             # share my beads writes to the fork's refs/dolt/data
git push                 # share my code to the fork's refs/heads/main

Why this order?

  1. git pull --rebase first so a concurrent git push from another session doesn't reject yours as non-fast-forward. Classic git hygiene, applies as normal.
  2. bd dolt push before git push because the beads push is independent of the code push — if the code push fails, the beads push has already shared your task state, which is the more time-sensitive of the two. Skipping this is how stranded beads happen: you close 10 issues, crash your session, and nobody else sees any of them.
  3. git push last because that's what everyone else is watching for code updates.

How the two refs stay separate

Dolt ships with its own remote protocol, but when you point it at a URL ending in .git it switches into git mode: it speaks the Git Smart HTTP protocol, uses your ordinary git credential helper, and pushes all its database state to a sibling ref called refs/dolt/data. bd dolt push is a thin wrapper around that.

$ bd dolt remote list
origin  git+https://github.com/idvorkin-ai-tools/idvorkin.github.io.git
#       ^^^^ dolt auto-added this prefix — URL ends in .git -> git mode

What the repo holds after a bd dolt push, asked via git ls-remote:

$ git ls-remote https://github.com/idvorkin-ai-tools/idvorkin.github.io.git
abc1234…  HEAD
def5678…  refs/dolt/data          <- beads task DB
abc1234…  refs/heads/main         <- code, PRs, branches
abc1234…  refs/heads/feature-X    <- more code branches
# ...

Aha. GitHub's file browser renders only refs/heads/*. If code on main hasn't changed, GitHub looks identical before and after a beads push — even though your task DB is completely different. A vanilla git clone fetches only refs/heads/* and misses beads entirely. You need bd dolt clone (or dolt clone directly) to rehydrate the task DB.

Why beads never flows to upstream

If you tried to run beads through the code workflow — "open a PR with my task changes, have Igor review and merge" — three things would break:

  1. bd dolt push doesn't update refs/heads/main. It only writes refs/dolt/data. A PR opened after a beads push would show identical refs/heads/main on both sides.
  2. GitHub's PR "Files changed" tab would be empty. The compare view diffs refs/heads/<base>..refs/heads/<head>. Both sides are the same commit → "nothing to compare." Igor has no visual diff of the task changes.
  3. The "Merge PR" button only advances refs/heads/main. It cannot move refs/dolt/data. Clicking Merge would change nothing about upstream's beads DB.

None of this is a bug — it's the intended behaviour. Beads is not supposed to flow to upstream. The fork is the authoritative task store; upstream should never see it.

Why agents don't collide on beads writes

Many concurrent writers would be unworkable with SQLite-in-git (unmergeable binary blob) or with a GitHub PR per task change (serialized, reviewed). Dolt's row-level three-way merge makes it routine:

  • Different rows: two agents creating different issues, or one closing bd-42 while another updates bd-99, merge automatically on the next bd dolt pull → bd dolt push.
  • Same row: two agents touching bd-123 at the same time produce a cell-level conflict stored as actual rows in dolt_conflicts_issues with base_*/our_*/their_* columns. You resolve via SQL or dolt conflicts resolve --theirs, commit the merge, push.

Scenarios

Seven reproducible shell scripts. Each wipes its own run directory, sets up fresh state, leaves artifacts for you to poke at. ./scripts/run-all.sh runs them end-to-end in ~20 seconds, offline. Scenarios use a generic items table but the mechanics are identical to what beads does with its issues table.

6. Why GitHub doesn't show beads commits — the dual-ref illusion
